# Layer 08 — Data Quality Optimization
> dbt schema tests (`_staging.yml`, `_core.yml`), a singular test
> (`dbt/tests/dim_drug_category_row_count.sql`), and Great Expectations
> (`scripts/run_ge_validation.py`). Theme: **test the right thing, at the right severity, with the
> right tool** — coverage as a KPI, not a 100% gate. See [00_INDEX.md](00_INDEX.md).

---

### DQ-01 — Two-tier testing: dbt for row-level, GE for table-level distributions  [✅ DONE]
- **Junior mistake:** Use one tool for everything (or rely on "the run didn't error").
- **Why it bites:** dbt generic tests are great at row-level constraints but awkward for table-level row counts and FK-resolution *rates*; using only one tool means one whole class of check goes missing.
- **Optimized (this repo):** dbt covers `not_null`/`unique`/`relationships`/`accepted_range`/`accepted_values`; GE covers row-count bands and resolution-rate distributions (the DQD KPIs). `scripts/run_ge_validation.py:2-9`, `dbt/models/staging/_staging.yml`, `dbt/models/marts/core/_core.yml`
- **Business one-liner:** "We check both individual records and dataset-wide health, using the right tool for each."
- **Soundbite:** *"Row-level integrity and table-level distributions are different questions. Use the tool built for each."*

### DQ-02 ★ — Coverage as an SLA band, not a 100% / exact gate  [✅ DONE]
- **Junior mistake:** Assert match rate `== 100%` (or any "perfect" target) on real-world dirty data.
- **Why it bites:** Honest partial matching (ADR-003) means 100% is impossible; the test would fail forever and get muted — killing the signal.
- **Optimized (this repo):** `ExpectColumnProportionOfNonNullValuesToBeBetween(drug_sk, 0.65, 1.0)` (measured 71.9%) and `condition_sk >= 0.90` (measured 98.9%) — a floor that protects quality without demanding the impossible. `scripts/run_ge_validation.py:73-80`
- **Business one-liner:** "We hold match coverage above a guaranteed floor and track it as a number we improve — not a perfect score we'd have to fake."
- **Soundbite:** *"A 100% test on dirty data isn't a quality gate, it's a snooze button waiting to happen."*
- **Related:** XWK-06, XWK-02, GLD-07 (honest-coverage thread)

### DQ-03 ★ — Row-count bounds that account for SCD2 growth  [✅ DONE]
- **Junior mistake:** `ExpectTableRowCountToEqual(133654)` — pin the exact current row count.
- **Why it bites:** `dim_drug` carries full SCD2 history; its row count is monotonically non-decreasing as snapshots accumulate, so an `==` baseline is green once and red on every run after the second.
- **Optimized (this repo):** `ExpectTableRowCountToBeBetween(min_value=133654, max_value=200000)` — a floor plus a sane ceiling, with the reasoning documented. `scripts/run_ge_validation.py:43-48`
- **Business one-liner:** "Our checks expect the history to grow over time, so they catch real anomalies instead of failing on normal accumulation."
- **Soundbite:** *"Don't assert equality against a number that's supposed to move. Bound it."*

### DQ-04 — Severity `warn` for legitimate-but-imperfect source, `error` for real defects  [✅ DONE]
- **Junior mistake:** Hard-`error` on every null, everywhere.
- **Why it bites:** 3 of 136,038 products (OTC sanitizers) genuinely ship with no generic name — a hard failure would block every build over legitimate source data, training people to ignore failures.
- **Optimized (this repo):** `generic_name` `not_null` is set to `severity: warn` with the reason documented, so the build proceeds and the signal stays meaningful. `dbt/models/staging/_staging.yml:26-32`
- **Business one-liner:** "Known harmless source quirks raise a warning, not a stop — so a real failure still means something."
- **Soundbite:** *"If everything is an error, nothing is. Reserve `error` for things that should actually block."*

### DQ-05 ★ — Conditional tests scoped with `where:` to the rows they apply to  [✅ DONE]
- **Junior mistake:** Blanket `not_null` on `match_confidence` across the whole dimension.
- **Why it bites:** `atc_category` seed rows have `match_confidence` null *by design* (GLD-02) — an unscoped test would fail on correct data and pressure someone to corrupt the column to make it pass.
- **Optimized (this repo):** the test is scoped `where: "drug_member_type = 'ndc_product'"`; likewise `dq_reason` is required only `where: "dq_flag = true"`. `dbt/models/marts/core/_core.yml:34-43`, `dbt/models/staging/_staging.yml:44-48`
- **Business one-liner:** "Each rule is applied only to the rows it should govern, so correct-by-design data never triggers a false alarm."
- **Soundbite:** *"A test that fails on correct data isn't strict — it's wrong. Scope it to where the rule holds."*

### DQ-06 — Referential integrity via `relationships`; nullable FKs tested as nullable  [✅ DONE]
- **Junior mistake:** Trust that joins line up; write no foreign-key tests.
- **Why it bites:** Orphan keys (a fact pointing at a missing dim row) get discovered in a dashboard, not in CI.
- **Optimized (this repo):** `relationships` tests on every fact→dim key — non-null for `fact_sales` (keys always resolve) and *nullable* for `fact_review.drug_sk`/`condition_sk` (partial-match policy), so integrity is enforced without forbidding legitimate nulls. `dbt/models/marts/core/_core.yml:54-65` vs `:86-95`
- **Business one-liner:** "Every link between facts and dimensions is verified automatically, while still allowing the honest 'not matched' cases."
- **Soundbite:** *"Test the foreign key — and test it as nullable exactly where nulls are allowed."*

### DQ-07 — Pin allowed value sets with `accepted_values` to catch drift  [✅ DONE]
- **Junior mistake:** No domain test on coded columns.
- **Why it bites:** A new or mistyped `atc_code`, `drug_member_type`, or match tier slips in silently and quietly skews every downstream rollup.
- **Optimized (this repo):** `accepted_values` pins the ATC set, `['ndc_product','atc_category']`, and the match-confidence vocabulary. `dbt/models/staging/_staging.yml:12-13`, `dbt/models/marts/core/_core.yml:32-33,40-43`
- **Business one-liner:** "If an unexpected category code ever appears, the build flags it instead of silently distorting the numbers."
- **Soundbite:** *"Pin the vocabulary. Unexpected values should fail a test, not a board meeting."*

### DQ-08 — Singular test for an exact business invariant  [✅ DONE]
- **Junior mistake:** Assume the `UNION ALL` that builds `dim_drug` produced the right number of seed rows.
- **Why it bites:** A duplicate or dropped seed row silently over/under-counts the category members, throwing off every `fact_sales` rollup — with no generic test that would catch it.
- **Optimized (this repo):** a singular test asserts `dim_drug` has exactly one `atc_category` member per seed code (returns 0 rows = pass). `dbt/tests/dim_drug_category_row_count.sql`
- **Business one-liner:** "A specific business rule — one category per seed — is checked exactly, so the sales dimension can't silently gain or lose categories."
- **Soundbite:** *"Generic tests catch generic problems. For a specific invariant, write the specific test."*
