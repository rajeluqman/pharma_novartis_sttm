## Company Background
**Helvetia Pharma AG** runs the **enVision** commercial analytics platform. Three project teams each
built a pipeline from a **different real data source, to a different standard**. The
**Architecture / STTM Lead** does not rebuild them — the Lead **reviews, reconciles, and
consolidates** their mappings, models, and architecture into one governed warehouse, and guarantees
the consolidated data is ready by **07:00**.

## Business Context
This role is **Data Governance / Metadata / Architecture Coordinator with a DE background** — not a
daily Spark coder. The day-to-day: take divergent team artifacts → review against the enVision
standard → resolve conflicts → consolidate → version → publish to Confluence → keep lineage honest →
guarantee the 7AM SLA on the consolidated DAG.

The pain point is **divergence**: three teams describe "a drug" three different ways, with three
grains, three naming conventions, and three separate STTM/model documents. There is **no shared join
key out of the box** — building the conformed crosswalk *is* the governance work.

## The Consolidation Substrate — Three Real Pharma Projects

Each is a thin mini-pipeline from a real dataset, built to a deliberately different standard. These
divergences are the material the Lead consolidates (the core JD test).

| Team | Real dataset | Source type | Grain | "Drug" identified by | Naming / date |
|------|-------------|-------------|-------|----------------------|---------------|
| **Alpha** — Commercial Sales | Kaggle `milanzdravkovic/pharma-sales-data` | CSV (POS time series) | day / hour × ATC category | **ATC code** (M01AB, N02BE…) | `UPPER_SNAKE`, `datum` |
| **Beta** — Product Master / Regulatory | **FDA NDC Directory** (openFDA) | JSON/CSV API (daily snapshot) | one row per product (SCD) | **NDC code + generic/brand name + `pharm_class`** | `lower_snake`, `marketing_start_date` |
| **Gamma** — Patient Voice / Digital | **UCI Drug Review (Drugs.com)** | CSV (review events) | one row per review event | **free-text `drugName` + `condition`** | `camelCase`, `reviewDate` |

**Built-in conflicts to resolve** (the interview drills, now backed by real data):
1. **No shared key** ⚠️ — Alpha=ATC, Beta=NDC+name+pharm_class, Gamma=free-text name. The Lead must build a conformed `dim_drug` **crosswalk** (ATC ↔ pharmacologic_class ↔ generic name ↔ free-text). This *is* the "Project A vs Project B define X differently" test — real.
2. **Grain**: day/hour vs product-snapshot vs review-event — STTM transforms + reconciliation rules.
3. **Naming/date**: `datum` vs `marketing_start_date` vs `reviewDate`; UPPER vs lower vs camel.
4. **SCD**: NDC products are added/withdrawn daily → SCD Type 2 on `dim_drug` (real, not contrived).

## Architecture & Layer Topology (4-tier, thick build)

```
 SOURCES (divergent)      LANDING          BRONZE         ENRICH (Silver)    GOLD (Data Mart + RRD)
 Alpha sales (CSV)   ─► landing/alpha ─► bronze.alpha ─► enrich.alpha ┐
 Beta  NDC (API)     ─► landing/beta  ─► bronze.beta  ─► enrich.beta  ┼─► STAR ───► OBT serving
 Gamma reviews (CSV) ─► landing/gamma ─► bronze.gamma ─► enrich.gamma ┘   dim_drug    obt_sales_wide
                        (immutable raw)  (+load meta)   (clean, per-src)   SCD2        obt_review_wide
 enVision:  Source   →  Landing        → Bronze        → Enrich          → Data Mart → RRD
```

- **Divergence** lives in Landing/Bronze/Enrich (3 separate sets — the substrate).
- **Convergence/governance** lives in Gold: one consolidated **star** (conformed `dim_drug` SCD2 +
  crosswalk) + **OBT** serving materialized from the star for 7AM BI. See ADR-001/002/003.
- **enVision vocab**: Landing→Bronze→Enrich→Data Mart→RRD — the 5-hop chain the STTM lineage maps to.
- **Build is THICK** (deliberate, for learning): full SCD2, incremental facts, partitioning/clustering,
  late-arriving + crosswalk exceptions — not minimal. Governance layer remains the showcase.

## Primary Business Questions (weighted to the JD)

### Group A — STTM Governance (≈40%) ⭐ core

#### G1: Can all three projects' columns consolidate into ONE STTM with 100% lineage?
- **Why it matters**: the #1 JD test — review Alpha+Beta+Gamma STTMs → consolidate → publish.
- **Required data**: each source schema → Enrich → Data Mart → RRD, with transform + business rule per column.
- **KPI**: **100%** of consolidated facts + Enrich columns mapped in one `docs/sttm/STTM.md`; **0** orphan/untraceable target columns; version-controlled + published to Confluence.

#### G2: How do you reconcile three definitions of "a drug" into one conformed dimension?
- **Why it matters**: the exact "Project A vs B define X differently" question — here it's real (ATC vs NDC/pharm_class vs free-text name).
- **Required data**: ATC categories (Alpha), NDC `pharm_class` + names (Beta), free-text `drugName` (Gamma); a crosswalk strategy.
- **KPI**: one approved `dim_drug` definition + crosswalk table; all three source identifiers mapped to it in STTM + `business_glossary`; coverage % of Gamma free-text names matched, with an unmatched-exceptions log.

#### G3: A field is wrong in the Data Mart — where do you investigate? (backward lineage)
- **Why it matters**: tests lineage reading, the daily reality of the role.
- **Required data**: full chain Source → Enrich → RRD → Data Mart for the suspect field.
- **KPI**: a documented backward-trace (Data Mart → RRD → Enrich → Source) isolating the layer at fault, citing the STTM row.

#### G4: A source schema changes (NDC adds/withdraws products) — how do you keep STTM accurate?
- **Why it matters**: STTM accuracy maintenance + version control under real change (NDC updates daily).
- **Required data**: a simulated/real NDC delta; impacted mappings.
- **KPI**: STTM updated + versioned (changelog), lineage re-validated, downstream impact noted before re-publish.

### Group B — Documentation Governance (≈30%) ⭐

#### D1: Consolidate three teams' architecture inputs into ONE Architecture Handbook.
- **Why it matters**: JD #2 — own the final AH, align to enVision standard, publish to Confluence.
- **Required data**: each team's architecture notes (layers, tools, assumptions).
- **KPI**: one approved `docs/architecture_handbook/AH.md`; Data Architect sign-off; published to the designated Confluence page with correct page hierarchy.

#### D2: Maintain version control + changelog discipline across AH and STTM.
- **Why it matters**: "maintain the latest approved copy" — version hygiene is explicit in the JD.
- **Required data**: doc version history, approval log.
- **KPI**: every published doc carries version + approver + date; only architecture-approved versions reach Confluence (governance gate).

### Group C — Data-Model Governance (≈20%)

#### M1: How do you ensure consistency across the three teams' data models?
- **Why it matters**: JD #3 — consolidate the Erwin model (Enrich / Data Mart / RRD), one governed model not three.
- **Required data**: each team's tables, keys, grain; naming standard; conformed dimensions.
- **KPI**: one Erwin-style ERD (`docs/erwin/ERD.md`) across all three layers; conformed `dim_drug`, `dim_date`, `dim_condition`; relationship validation passes.

#### M2: How do you model NDC products that are added/withdrawn over time?
- **Why it matters**: tests SCD reasoning — and NDC genuinely changes daily, so it's real.
- **Required data**: NDC marketing_start/end dates; grain of `dim_drug`.
- **KPI**: SCD Type 2 on `dim_drug` with effective dating; documented in DATA_MODEL.md + ADR.

### Group D — DAG Analysis (≈10%)

#### S1: Can the consolidated daily pipeline finish before 07:00, and where is the bottleneck?
- **Why it matters**: JD #1 — 7AM SLA via end-to-end DAG analysis.
- **Required data**: Airflow DAG run history (local) / MWAA logs (cloud), task-level Gantt, critical path.
- **KPI**: end-to-end runtime vs **240-min budget** (03:00 start → 07:00 SLA); critical-path minutes; before/after per fix logged in `docs/sla/SLA_ANALYSIS.md`.

> **Chain**: build 3 thin pipelines (substrate) → **G1–G4** consolidate + defend the STTM/crosswalk →
> **D1–D2** consolidate + version + publish docs → **M1–M2** consolidate + govern the model →
> **S1** guarantee it lands by 7AM. Weighting mirrors the JD, not a coding role.

## Datasets (all real, all verified)

### Project Alpha — Commercial Sales
- **Source**: Kaggle — Pharma Sales Data (point-of-sale, single pharmacy).
- **URL**: https://www.kaggle.com/datasets/milanzdravkovic/pharma-sales-data
- **Files**: `salesdaily.csv`, `saleshourly.csv`, `salesweekly.csv`, `salesmonthly.csv`.
- **Grain/keys**: day|hour × ATC category; 8 ATC columns (M01AB, M01AE, N02BA, N02BE, N05B, N05C, R03, R06); `datum`, `Year`, `Month`, `Hour`, `Weekday Name`.

### Project Beta — Product Master / Regulatory
- **Source**: FDA **National Drug Code (NDC) Directory** via openFDA (updated daily).
- **URL**: https://open.fda.gov/apis/drug/ndc/download/  (also NBER CSV mirror: https://www.nber.org/research/data/national-drug-code)
- **Grain/keys**: one row per product; `product_ndc`, `proprietary_name`, `generic_name`, `dosage_form`, `route`, `pharm_class`, `labeler_name`, `marketing_start_date`/`marketing_end_date` (→ SCD2).
- **License**: US public domain (FDA). OPENFDA_API_KEY optional (higher rate limit).

### Project Gamma — Patient Voice / Digital
- **Source**: UCI **Drug Review Dataset (Drugs.com)** — patient reviews/ratings.
- **URL**: https://www.kaggle.com/datasets/jessicali9530/kuc-hackathon-winter-2018  (UCI: https://archive.ics.uci.edu/dataset/462/drug+review+dataset+drugs+com)
- **Grain/keys**: one row per review; `drugName` (free text), `condition`, `rating` (1–10), `date`, `usefulCount`. ~215k rows.
- **License**: CC BY 4.0.

> **Linkage note (the governance work):** there is **no native shared key** across the three. The
> Lead defines a conformed `dim_drug` and builds the crosswalk: ATC (Alpha) ↔ `pharm_class`/generic
> name (Beta) ↔ free-text `drugName` (Gamma). Expect partial match coverage — log unmatched as STTM
> exceptions. `dim_date` is the clean conformed dimension across all three.

## Constraints
- **Stack (locked, HYBRID)**:
  - *Dev (local, free)*: Docker Airflow (localhost:8080) · DuckDB · dbt Core · Great Expectations · dbdiagram.io (Erwin clone) · Markdown-as-Confluence.
  - *Deploy (cloud, short window)*: **AWS MWAA** (Managed Airflow) + **Snowflake** + **dbt** → for portfolio artifacts/screenshots, then teardown.
- **Budget**: MWAA is not free-tier, but a **same-day create→run→teardown spike ≈ $3–5** (the ~$350/mo figure only applies if left running 24/7). Develop fully on aws-mwaa-local-runner (free), then do ONE short real-MWAA window for the artifact and teardown same day. Snowflake = 30-day trial ($400 credits). Everything else free/local.
- **Timeline**: ongoing gym (substrate build → governance consolidation → SLA), not a 1-day sprint.

## Success Criteria
- **Substrate**: three thin pipelines (Alpha/Beta/Gamma) produce divergent outputs — deliberately minimal.
- **STTM (40%)**: one consolidated, version-controlled `STTM.md`, 100% coverage, conflicts + crosswalk resolved, published.
- **Docs (30%)**: one consolidated `AH.md`, versioned, architecture-approved, published to Confluence.
- **Model (20%)**: one Erwin ERD across Enrich/Data Mart/RRD; conformed `dim_drug`/`dim_date`/`dim_condition`; SCD2 on `dim_drug`.
- **DAG (10%)**: consolidated daily pipeline meets 7AM SLA; bottleneck fix logged with before/after runtime.
- **Cloud artifact**: one successful MWAA + Snowflake run captured (screenshot/log) for the portfolio, then torn down.
- README with a business Q&A table + `INTERVIEW_GUIDE.md` covering Tier-1 drills (STTM, lineage, modelling, DAG).

## Out of Scope (do NOT do these)
- Heavy hands-on Spark optimization as the showcase — build is substrate, not the deliverable.
- Leaving MWAA/Snowflake running beyond the artifact window (cost).
- Real patient-identifiable data — Gamma reviews are public/aggregate; no PII linkage.
- ML/NLP on reviews as a deliverable — Gamma is a *governance source*, sentiment modelling is out.
- Real Confluence/Erwin/enVision licenses — Markdown + dbdiagram.io clones.
